HTTP权威指南（一）：HTTP 概述

HTTP 是什么？

HTTP 是一种应用层协议，用于沟通 web 浏览器以及服务器，让用户能够足不出户的访问互联网上的各种资源，是现代互联网的基础。

HTTP 使用 TCP 来进行连接，可以保证数据传输的顺序以及完整性。

web 浏览器端和服务器是如何通信的？

web 浏览器端和服务器通过 HTTP 请求来互相通信。

什么是 MIME？

MIME 的全称是 Mutiple Internet Mail Extensiton ，翻译为 媒体类型 。

在互联网上的文件有文本、图片、视频等很多种类型，在 web 端和服务器端进行通信时，服务器会为所有 HTTP 数据对象（body）附加一个 MIME 类型（content-type），使得 web 端能够区分不同类型文件。

MIME 是一串字符串，由主要对象类型和子类型构成，中间通过斜杠进行区分。如 json 类型可以表示为 application/json；普通的 HTML 文本类型可以表示为 text/html 等。

URI vs URL vs URN？

URI（Uniform Resource Identifier，统一资源标识符） 是一个资源在整个互联网范围内的唯一标识，用于指示并定位资源。URI 有两种形式，分别是 URL 和 URN。

URL（Uniform Resource Locator，统一资源定位符） 是 URI 最常见的表示形式，它说明了一个资源在某台特定服务器上的位置。

URL 通常由三部分构成：

  • 协议头（scheme），表明资源应该通过哪种协议类型访问。最常见的就是 http:// 这一协议头。
  • 服务器的因特网地址。可以是域名，也可以是 IP 地址。如 www.baidu.com 。
  • 剩下的部分指明了资源位于该服务器上的位置。如 /dist/bubble.jpg。

如今，几乎所有的 URI 都是 URL。

URI（Uniform Resource Name，统一资源名）是作为特定资源的唯一名称使用的。通过使用这种与位置无关的表述，让我们得以把资源四处搬运存储。同时 URN 还是与协议无关的，也就是说我们可以通过不同的协议访问同一个资源。

URN 目前还在实验阶段，并未大规模使用。以目前的网络架构，并不支持这种分布式解析资源位置的服务。

什么是 HTTP 事务？

一个 HTTP 事务是由一个客户端发送的 HTTP 请求和一个服务器发送的 HTTP 响应结果组成的。这种通信是通过名为 HTTP 报文（HTTP Message） 的格式化数据块进行的。

在客户端发往服务器的报文中，会指定一个 HTTP 方法（HTTP Method）。HTTP 方法用于指定服务器收到报文后，应该进行的操作。常见的 HTTP 方法有：GET、POST、DELETE、PUT 和 HEAD。

💡 PUT vs POST

二者都用于向服务器提交数据。其中，PUT 会将来自客户端的数据保存为文件；POST 则常常用于传递信息，如提交表单等。
二者最根本性的区别在于是否幂等。所谓幂等，就是指相同的请求，无论执行多少次，结果都是一样的（优点类似于纯函数）。
一般来说，PUT 是幂等方法，POST 不是。当然实现上可能会有偏差，你要非把 PUT 变成非幂等的也没人能说啥，但至少不太符合直觉和 RestFul API 的设计规范。

在服务器发往客户端的响应报文中，会携带一个状态码，告知客户端请求是否成功或者是否需要进行进一步的操作。常见的状态码：200、301、302、404、500。

HTTP 报文的结构？

HTTP 报文分为请求报文和响应报文，二者的格式是相同的。HTTP 报文都是字符串，可以很方便的进行读写。

HTTP 报文分为三个部分：

  • 起始行：报文的第一行就是起始行。起始行一般包括 HTTP 方法、HTTP 版本、请求链接以及状态码等信息。
  • 首部字段：紧接着起始行的，是首部字段。每个 HTTP 报文都可以有零个或多个首部字段。在首部字段的最后，是一个空行，表示首部字段已结束。
  • 主体：在空行之后，便是报文主体部分。其中，请求报文主体中会包含要向服务器传递的数据，响应报文主体中会包含要返回给客户端的数据。与起始行和首部不同，主体可以包含任意格式的数据。

HTTP 传输的步骤？

HTTP 是应用层协议，通过 TCP（传输层）/ IP（网络层）来传输数据。

HTTP 传输的步骤为：

  1. 客户端解析 URL，获取目标主机名（域名）。
  2. 客户端通过 DNS 解析域名，获取目标主机的 IP 地址。
  3. 客户端将端口号（如果有的话）从 URL 中解析出来（TCP 默认为 80 端口）。
  4. 客户端与目标主机建立 TCP 连接。
  5. 客户端向目标主机发送 HTTP 请求报文。
  6. 服务器向客户端发送 HTTP 响应报文。
  7. 客户端处理响应报文，展示内容。

上面的只是简略版，省去了 N 多细节，比如 TCP 连接如何建立，TCP 连接的复用等，在这里就不展开了。

HTTP 协议有哪些版本？

HTTP 有 5 个大版本，按照时间排序分别为：

  • HTTP/0.9：设计有很大缺陷，如只支持 GET 方法、不支持多媒体内容等。因此退出不就就被 HTTP/1.0 取代了。
  • HTTP/1.0：第一个广泛应用的版本。与 HTTP/0.9 相比，增加了诸如 HTTP 版本号、各种头部字段、新的请求方法等若干功能。
  • HTTP/1.0+：添加了许多新特性，如 keep-alive、代理连接等。这些功能虽然得到了广泛应用，但并未被纳入到官方标准中，因此被称为 HTTP/1.0+。
  • HTTP/1.1：目前被广泛应用的版本。在 HTTP/1.0 基础上做了一些修正。
  • HTTP-NG（HTTP/2.0）：尚未应用，未来可期。

常见的构成互联网的程序

鉴定网络热门程序的视频（小亮口吻

  • 代理：转发请求。位于客户端和服务器之间的程序。代理既可以作为客户端向服务器发送请求，也可以作为服务器向客户端发送响应。
  • 缓存：资源缓存服务器，将距离较远的服务器上的资源缓存下来，使得在客户端请求资源时，不必从距离很远的服务器上取得。
  • 网关：一般主要职责是协议转换，如将 HTTP 转换为 HTTPS。
  • 隧道：隧道是建立起来之后，就会在两条连接之间进行盲转发的 HTTP 程序。这种程序会在第 8 章中细讲。
  • Agent 代理：所有发送 web 请求的应用程序都是 Agent 代理，如浏览器、爬虫等。

需要说明的是，虽然这些程序的职责各有不同，但程序彼此之间却并没有十分明显的职责界限，一个程序常常可以担任多种工作，强行区分并没有太大意义。

除此之外，还有一些常见程序没有列举出来，如反向代理等，这里不做赘述。

补充知识：Telnet 和 netcat

Telnet 是用于模拟 HTTP 客户端的程序，但其并不能作为服务器使用。

netcat 是可操作 TCP、UDP 和 HTTP 流量的程序，且可为其编写脚本。
